# Рендеринг

> Один кадр: обновить логику (ticker) → renderer рисует `stage` → GPU compositing в canvas.

## Цикл кадра

```
Ticker callback (логика, анимации)
        ↓
renderer.render(stage)  // или app.render() в простых случаях
        ↓
Обход scene graph → сборка batch’ей → draw calls → canvas
```

```ts
app.ticker.add((ticker) => {
  hero.x += 2 * ticker.deltaTime;
  // Явный render обычно не нужен — Application рендерит после ticker
});
```

`autoStart` у ticker по умолчанию включён; можно рендерить вручную через `app.renderer.render(app.stage)`.

## Renderer: WebGL / WebGPU / Canvas

| Backend | Когда |
|---------|--------|
| **WebGPU** | Современные браузеры, `preference: 'webgpu'` |
| **WebGL** | Fallback, широкая поддержка |
| **Canvas** | Редко, простые сцены или отладка |

```ts
await app.init({
  preference: 'webgl', // 'webgpu' | 'webgl' | 'canvas'
  antialias: true,
  resolution: window.devicePixelRatio,
  autoDensity: true, // CSS размер canvas ≠ внутреннее разрешение
});
```

**`resolution` + `autoDensity`** — важная пара на Retina: логические 800×600, физически canvas больше в пикселях.

## Batching (батчинг)

Рендерер **объединяет** спрайты с одной текстурой (и совместимым blend/state) в один draw call.

**Ломает batch (увеличивает draw calls):**

- смена текстуры между соседними спрайтами;
- **фильтры** на объекте (отдельный проход);
- **маски**;
- разный blend mode;
- `Graphics` / сложный mesh — отдельные пайплайны.

**На собесе:** «1000 спрайтов с одним атласом ≈ мало draw calls; 1000 разных текстур ≈ до 1000 draw calls».

## Culling

Объекты вне экрана могут не попадать в рендер (зависит от настроек/bounds). Для больших миров — ручной culling по `getBounds()` или spatial hash.

## RenderTexture

Рендер сцены/контейнера в текстуру (мини-карта, post-processing, offscreen):

```ts
import { RenderTexture } from 'pixi.js';

const rt = RenderTexture.create({ width: 256, height: 256 });
app.renderer.render({ container: subScene, target: rt });
// rt — текстура для другого Sprite
```

Не забыть `rt.destroy()` когда не нужна.

## Частые вопросы

**Почему падает FPS?**  
Много draw calls, фильтры, overdraw, большие текстуры, лишние `render()` в секунду, утечки GPU.

**Double buffering?**  
Браузер композитит canvas; Pixi рисует в WebGL framebuffer, итог — на экране через compositor.

**Нужно ли вызывать `render` каждый кадр?**  
С `Application` + ticker — обычно нет, фреймворк сам перерисовывает при изменениях сцены.

## Шпаргалка

> Pixi рисует дерево `stage` через WebGL/WebGPU, группируя спрайты с одной текстурой в batch. Draw call дорогой — атласы и порядок детей важны. Фильтры и маски — отдельные проходы. `resolution`/`autoDensity` — для чёткости на HiDPI.
