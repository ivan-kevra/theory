# Как работает очистка памяти в PixiJS

> Актуально для **PixiJS v8**. GPU-память (текстуры, буферы, шейдеры) JS-сборщик мусора **не освобождает** — нужны явные вызовы или встроенные GC-системы рендерера.

## Почему это важно на собесе

- Обычный `sprite = null` **не освобождает** GPU-ресурсы, пока объект не уничтожен через `destroy()` или пока не сработает автоматический GC Pixi.
- Текстуры часто **шарятся** между спрайтами — неверный `destroy({ texture: true })` может сломать другие объекты.
- Утечки проявляются как рост VRAM/RAM и просадки FPS при длительной работе SPA/игры.

---

## 1. Явная очистка: `destroy()`

Основной способ — вызвать `destroy()` на `Container`, `Sprite`, `Text`, `Graphics`, `Application` и т.д.

```ts
import { Sprite } from 'pixi.js';

const sprite = new Sprite(texture);

// Только объект; текстура остаётся в кэше/GPU для переиспользования
sprite.destroy();

// Объект + дети + текстура + источник текстуры (освобождает GPU)
sprite.destroy({
  children: true,
  texture: true,
  textureSource: true,
  context: true,  // canvas-контекст у Graphics/Text
  style: true,    // стили текста
});
```

| Опция | Что делает |
|--------|------------|
| `children: true` | Рекурсивно уничтожает дочерние display objects |
| `texture: true` | Уничтожает `Texture`, привязанную к объекту |
| `textureSource: true` | Уничтожает источник (`TextureSource`) — **осторожно**, если один source у нескольких текстур |
| `context` / `style` | Освобождает ресурсы у Graphics и Text |

**Правило:** если текстура ещё нужна другим спрайтам — **не** передавай `texture: true` / `textureSource: true`.

При завершении приложения:

```ts
await app.destroy(true, { children: true }); // true — удалить canvas из DOM
```

---

## 2. Ручная выгрузка текстур: `texture.source.unload()`

Когда автоматического GC недостаточно (много динамически подгружаемых ассетов):

```ts
import { Assets } from 'pixi.js';

const texture = await Assets.load('hero.png');

// ... использование ...

texture.source.unload(); // снимает данные с GPU, объект texture может остаться
```

Для ассетов из `Assets`:

```ts
await Assets.unload('hero.png'); // убирает из кэша + выгружает GPU-ресурс
```

---

## 3. Автоматический GC текстур: `TextureGCSystem`

Включён по умолчанию в `Application`. Следит за **неиспользуемыми** текстурами и удаляет их с GPU.

| Параметр | По умолчанию | Смысл |
|----------|--------------|--------|
| `textureGCActive` | `true` | Вкл/выкл систему |
| `textureGCMaxIdle` | `3600` кадров | Сколько кадров текстура может простаивать (~60 с при 60 FPS) |
| `textureGCCheckCountMax` | `600` кадров | Как часто проверять (~10 с при 60 FPS) |

```ts
import { Application } from 'pixi.js';

const app = new Application();
await app.init({
  textureGCActive: true,
  textureGCMaxIdle: 7200,
  textureGCCheckCountMax: 1200,
});
```

**Важно:** пороги считаются в **кадрах**, не в секундах. При 30 FPS текстуры живут дольше в «реальном» времени.

Это **не замена** `destroy()` — отложенная подстраховка для забытых ссылок на текстуры.

---

## 4. GC рендерируемых объектов: `RenderableGCSystem`

Страховка для ситуации «спрайт создали, отрисовали, ссылку потеряли, `destroy()` не вызвали»:

- По умолчанию проверка ~каждые **30 с**
- Ресурсы, не попадавшие в рендер **~1 минуту**, очищаются с GPU
- При повторном использовании типа объекта GPU-ресурсы могут взяться из пула

```ts
// Плохо: утечка до срабатывания GC
let sprite = new Sprite(texture);
renderer.render(sprite);
sprite = null;

// Хорошо
sprite.destroy({ texture: true, textureSource: true });
sprite = null;
```

Отключение (если нужен полный ручной контроль): `renderableGCActive: false` при инициализации рендерера.

> В v8.15+ часть API `RenderableGCSystem` помечена deprecated — следи за changelog при апгрейде.

---

## 5. Что ещё чистить вручную

| Что | Как |
|-----|-----|
| Ticker / game loop | `app.ticker.remove(fn)` или `app.ticker.destroy()` |
| Слушатели событий | `obj.off('event', handler)` / `obj.removeAllListeners()` |
| `Assets` cache | `Assets.unload()` / `Assets.reset()` |
| Пул объектов | Переиспользование вместо `new Sprite` на каждый выстрел/партикл |

## 6. Типичные ошибки (частые вопросы на интервью)

1. **Удалили из сцены, но не `destroy()`** — `parent.removeChild(sprite)` только отвязывает от дерева, GPU-память остаётся.
2. **`destroy({ texture: true })` на shared texture** — другие спрайты получат битую/пустую текстуру.
3. **Массовый `destroy` в один кадр** — фриз; лучше батчить (например, 5–10 объектов за кадр).
4. **Путают JS heap и GPU memory** — DevTools покажут не всё; смотреть GPU/профайлер браузера.
5. **Забыли `Assets.unload`** — текстура остаётся в кэше Pixi даже после `destroy` спрайта.

---

## 7. Краткая схема

```
Создание Sprite/Texture
        │
        ▼
   Использование в сцене
        │
        ├─► Готово навсегда ──► destroy({ children, texture, textureSource })
        │
        ├─► Временная текстура ──► texture.source.unload() / Assets.unload()
        │
        └─► Забыли destroy ──► TextureGCSystem + RenderableGCSystem (с задержкой)
```

---

## Шпаргалка для ответа устно

> В PixiJS память на GPU не собирается GC JavaScript. Мы явно вызываем `destroy()` с нужными флагами, для загрузок через `Assets` — `unload()`. Встроенные `TextureGCSystem` и `RenderableGCSystem` подчищают неиспользуемые текстуры и renderable-ресурсы по таймауту в кадрах, но это safety net, а не основная стратегия. Shared-текстуры уничтожаем только когда уверены, что больше нигде не используются.
